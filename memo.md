# Git ハンドブック（骨子）

Git の基本概念とよく使う操作を、はじめての人でも順を追って理解できるようにまとめるための骨子。各節は「概念 → なぜ必要か → 使い方」の流れで構成する。

---

## 1. Gitとは

- バージョン管理システム (Version Control System - VCS) の1つで、ソースコードの変更履歴を追跡・管理するツール。
- Gitの作者は、Linuxの生みの親としても有名なリーナス・トーバルズ（Linus Torvalds）。フィンランド出身のソフトウェアエンジニアで、2005年にLinuxカーネルの開発のために作成した。
- GitHub、GitLab、Bitbucket、CodeCommitなどのホスティングサービスと組み合わせることで、チームでの共同開発やコードレビューが効率化される。

### Tips
- Git以外のVCSとしては、SubversionやMercurialなどが存在するが、現在はGitが業界標準として広く使われている。

---

## 2. 基本概念

- リポジトリ
    - リポジトリとは
        - ファイルの変更履歴を保存する保管庫。プロジェクトのファイル群と、それらの全変更履歴（コミット）をまとめて管理する単位。
    - ローカルリポジトリ
        - 自分のPC上にあるリポジトリ。ここで編集・コミットを行う。ネットワークがなくても操作できる。
    - リモートリポジトリ
        - サーバー上（GitHub等のホスティングサービス）にあり、チームで共有するリポジトリ。ローカルとの間で変更をやり取りする。
- ワーキングツリー（作業ツリー）
    - ワーキングツリーとは
        - 実際にファイルを編集する作業スペース。リポジトリから取り出した、いま手元で見えているファイル群のこと。
- ステージング（インデックス）
    - ステージングとは
        - 次のコミットに含める変更を一時的に登録しておく場所。`git add` で変更をここに追加する。
- コミット
    - コミットとは
        - ステージングに登録した変更を、履歴として1つの区切りで記録する操作、またその記録単位。その時点のスナップショットを保存する。
    - なぜステージングを使用するのか
        - 変更したファイルの中から「今回のコミットに含めるもの」だけを選んで記録できるため。関連する変更ごとに意味のある単位でコミットを分けられる。
- ブランチ
    - ブランチとは
        - 履歴を枝分かれさせて、他に影響を与えずに独立して作業するための仕組み。並行して複数の作業を進められる。
    - なぜブランチを使用するのか
        - メインの履歴を安定させたまま、機能追加やバグ修正を分離して進めるため。
        - メインブランチだけでチーム開発すると起こり得ること
            - 未完成のコードがメインに混ざる、複数人の変更が衝突する、不具合の切り分けが難しくなる、など。
    - マージとは
        - 別々のブランチで行った変更を1つに統合する操作（詳細は「7. マージについてもっと知る」）。
    - 開発の大まかな流れ (CLI操作)
        1. メインブランチ（main / master）から作業用ブランチを作成する
        2. 作業用ブランチで編集・コミットを繰り返す
        3. リモートリポジトリにローカルリポジトリの変更内容を反映した作業用ブランチをプッシュする
        4. プルリクエストを作成し、コードレビューを受ける
            - プルリクエストとは
                - 「自分のブランチの変更をメインブランチに取り込んでほしい」という依頼。レビューや議論、承認のための仕組みで、多くはホスティングサービス上で行う。
        5. レビュー承認後、メインブランチにマージする

## 3. ワーキングツリーについてもっと知る
- ファイル追跡機能について
- 追跡状態の種類
- 追跡させない方法
- ワーキングツリーの操作に使うコマンド一覧
 - git status
 - git diff
 - git checkout
 - git add
 - git restore
 - git stash
 - git rm


- ファイルのトラッキング
    - tracked / untracked（ファイルの追跡状態）とは
        - Gitが変更を監視している状態を tracked、監視対象外を untracked と呼ぶ。一度 `git add` されたファイルは tracked になる。
        - ※「5. ブランチについてもっと知る」で扱う upstream branch / remote-tracking branch とは別の概念。
    - ファイルの状態の種類（tracked の内訳と untracked）
        - untracked（未追跡）: まだ一度も `git add` されておらず、Gitが変更を認識していない新規ファイル
        - modified（変更済み）: 追跡対象で、前回のコミットから内容が変更されているが、まだステージングされていない状態
        - staged（ステージ済み）: 変更が `git add` され、次のコミットに含める準備ができた状態
        - unmodified（変更なし）: 追跡対象で、前回のコミットから内容が変わっていない状態
        - ignored（無視）: `.gitignore` の設定によりGitが意図的に無視しているファイル
        - ※ 現在の状態は `git status` で確認できる。
- 変更を取り消す方法（`git restore <file>` / `git checkout -- <file>`）
    - ワーキングツリーの変更を破棄し、直前のコミット状態に戻す。
    - `git restore` は変更取り消し専用の新しいコマンドで、現在はこちらが推奨される。`git checkout -- <file>` は従来からある同等の方法。
- ステージングエリアに追加する方法（`git add <file>`）
    - 追加するタイミング
        - 変更が1つの意味あるまとまりになり、コミットに含めたいと判断できたとき。`git add -p` で変更の一部だけを選んで追加できる。
- 追跡させない方法
    - `.gitignore` ファイル
        - `.gitignore` とは
            - Gitに追跡させたくないファイルやディレクトリのパターンを記述するファイル。ここに書いたものは untracked として扱われず、`git status` にも現れない。
            - ※ すでに追跡済みのファイルは `.gitignore` に追記しても追跡対象のまま。除外するには `git rm --cached <file>` が必要。
        - よく除外されるファイルの例（`node_modules/`、`.env`、ビルド成果物など）
- 一時的に変更を退避させる方法（`git stash` / `git stash pop`）
    - `git stash` で現在の作業中の変更を一時退避し、クリーンなワーキングツリーに戻す。
    - `git stash pop` で退避した変更を再適用し、退避リストから取り除く（残したまま適用する場合は `git stash apply`）。
    - 別ブランチに切り替える前や、緊急の修正対応が必要なときなどに活用される。

## 4. コミットについてもっと知る
- コミットが持つ情報
- コミットするタイミング
- コミット履歴を確認する方法
- コミットを修正・操作する方法
- コミットの操作に使うコマンド一覧
    - git commit
    - git log
    - git show
    - git revert
    - git reset
    - git commit --amend


- コミットが持つ情報
    - コミットハッシュ（既定はSHA-1）: コミットを一意に識別するハッシュ値
    - 作者情報（Author）: コミットを作成したユーザー名とメールアドレス
    - 日時（Date）: コミットが作成された日時
    - コミットメッセージ: 変更内容を説明するメッセージ
    - 親コミットへの参照: 1つ前のコミットのハッシュ値（マージコミットの場合は複数）
    - スナップショット（ツリー情報）: その時点のファイル全体の状態
- コミットするタイミング
    - 1つの意味あるまとまり（機能追加、バグ修正など）ができた区切りごと。小さく頻繁にコミットすると、後から履歴を追いやすく、取り消しもしやすい。
- コミット履歴を確認する方法（`git log`）
    - `git log --oneline` で簡潔な一覧表示が可能
    - `git log --graph` でブランチの分岐・マージをグラフ形式で確認できる
- コミットを修正・操作する方法
    - 直前のコミットを修正する（`git commit --amend`）
        - コミットメッセージの修正や、ファイルの追加・変更をやり直す際に使用する。
        - ※ コミットを作り直すため、すでにプッシュ済みのコミットに使うと履歴が食い違う点に注意。
    - コミットを取り消す（`git revert` / `git reset`）
        - `git revert <hash>`: 指定したコミットの変更を打ち消す新しいコミットを作成する（履歴を残したまま取り消す）。実行にはワーキングツリーをクリーンにしておく必要がある。
        - `git reset --soft <hash>`: 指定したコミットまでHEADを戻す。ステージング（index）と作業ツリーは変更しないため、戻した分の変更はステージに残る。
        - `git reset --mixed <hash>`（既定）: 指定したコミットまでHEADを戻し、ステージング（index）もその状態に合わせる。作業ツリーの変更は残る（ステージから外れた状態になる）。
        - `git reset --hard <hash>`: 指定したコミットまでHEAD・ステージング・作業ツリーの変更をすべて破棄する。
        - ※ `revert` は履歴を残す安全な取り消し。`reset`（特に `--hard`）は履歴を書き換えるため、共有済みのコミットには使わない。

## 5. ブランチについてもっと知る

- ブランチのトラッキング
    - upstream branch を track するとは
        - ローカルブランチが特定のリモートブランチと紐づいている状態。紐づけ先を upstream branch と呼び、その関係を track と言う。
        - リモート側の状態を映す参照そのもの（例: `origin/main`）は remote-tracking branch と呼ぶ。
        - ※「3. ワーキングツリー」で扱う tracked / untracked（ファイルの追跡状態）とは別概念。
        - upstream branch が設定されていると、`git push` や `git pull` で対象リモートブランチを省略できる。
        - `git branch -vv` で各ブランチの upstream 設定を確認できる。
- ブランチを切り替える方法（`git switch <branch>` / `git checkout <branch>`）
    - `git switch` はブランチ切替専用の新しいコマンドで、現在は推奨。`git checkout` は従来からある同等の方法。
- ブランチを作成する方法とタイミング
    - `git switch -c <branch>`（または `git checkout -b <branch>`）で作成と同時に切り替えられる。
    - タイミング: 新しい機能追加やバグ修正など、独立した作業を始めるとき。メインブランチから分岐して作る。
- ブランチを削除する方法とタイミング
    - `git branch -d <branch>` でマージ済みブランチを削除（未マージを強制削除する場合は `-D`）。
    - タイミング: 作業が完了しメインブランチにマージされた後。役目を終えたブランチを整理する。
- ブランチの一覧を確認する方法
    - `git branch` でローカル、`git branch -a` で remote-tracking branch を含めて一覧表示する。
- ブランチ名を変更する方法
    - `git branch -m <old> <new>`（現在のブランチ名を変える場合は `git branch -m <new>`）。
- ポインタとは
    - ブランチの正体は「特定のコミットを指す軽量な参照（ポインタ）」。コミットするたびに、そのブランチのポインタが新しいコミットへ進む。
- HEADとheadとは
    - HEAD: 「いま自分がいる位置」を指す特別なポインタ。通常は現在チェックアウトしているブランチを指す。
    - head（小文字）: 各ブランチの先端（最新コミット）を指す参照の総称。実体は `refs/heads/` 以下に置かれ、ブランチ1つにつき1つの head がある。

## 6. リポジトリについてもっと知る
- 初期化とは
- クローンとは
- プッシュとは
- origin とは
- フェッチとは
- プルとは
- リポジトリ操作のコマンド一覧





- `git remote` コマンド
    - リモートリポジトリの一覧を確認する（`git remote -v`）
    - リモートリポジトリを追加する（`git remote add <name> <url>`）
    - リモートリポジトリを削除する（`git remote remove <name>`）
    - ※ 慣例として、既定のリモート名には `origin` が使われる。
- プッシュ（`git push`）とは
    - ローカルリポジトリの変更内容をリモートリポジトリに反映する操作。
    - 初回プッシュ時は `-u` オプションで upstream branch の紐づけを同時に行うことが多い（`git push -u origin <branch>`）。
- 強制プッシュ（`git push --force`）は原則禁止
    - リモートの履歴を上書きするため、他人がすでに取得した履歴を壊す恐れがある。共有ブランチ（特に main）では基本的に禁止とする運用が多い。
    - rebase や `git commit --amend` で履歴を書き換えた自分専用ブランチなど、やむを得ない場合は `--force-with-lease` を使う。他人の更新を検知すると上書きを中止できる。
    - main などの重要ブランチは、リモート側の branch protection で force push 自体を禁止しておくのが一般的。
- フェッチ（`git fetch`）とは
    - リモートリポジトリの最新情報をローカルに取得するが、ワーキングツリーには反映しない。取得した内容は remote-tracking branch（例: `origin/main`）で確認できる。
- プル（`git pull`）とは
    - `git fetch` の後に取得した変更を統合する操作。既定ではマージで統合する。
    - `git pull --rebase` を使うと、取得した変更をリベースで統合し履歴を一直線に保てる。

## 7. マージについてもっと知る

- マージとは
    - 異なるブランチの変更内容を統合する操作。
- マージの種類
    - Fast-forwardマージ: 分岐後にメインブランチへの変更がない場合、ポインタを進めるだけで統合する（マージコミットは作られない）。
    - Non Fast-forwardマージ（3-wayマージ）: 双方に変更がある場合、新たなマージコミットを作成して統合する。
    - Squashマージ: featureブランチの複数コミットを1つにまとめてから統合する方法。
        - `git merge --squash <branch>` の後に `git commit` する。マージコミットにはならず、親が1つの通常コミットになる。
        - 細かいコミットをメイン側では1つに集約でき、履歴を簡潔に保てる。GitHub / GitLab などの PR でも「Squash and merge」として選べる。
        - 元のブランチのコミット履歴はメインには残らない点に注意。
- コンフリクト（競合）とは
    - 同じファイルの同じ箇所が異なるブランチで変更された場合に発生する。Gitが自動で統合できず、手動での解決が必要になる。
    - コンフリクトの解消方法
        1. `git status` で競合しているファイルを確認する。
        2. ファイル内の競合マーカー（`<<<<<<<` / `=======` / `>>>>>>>`）を見て、正しい内容に手で編集する。
        3. 解決したファイルを `git add` でステージングする。
        4. マージを完了させる（マージ時は `git commit`、リベース時は `git rebase --continue`）。
- リベース（`git rebase`）とは
    - ブランチの派生元を別のコミットに付け替えることで、履歴を一直線に整える操作。
    - マージとリベースの使い分け
        - マージ: 分岐と統合の履歴をそのまま残したいとき。安全で、共有済みブランチにも使える。
        - リベース: 履歴を直線的に整えたいとき。ただしコミットを作り直すため、すでに共有（プッシュ）済みのコミットには使わない。

## 8. その他の便利な操作や補足

- タグ（`git tag`）とは
    - 特定のコミットに名前をつける参照。リリースバージョン（例: `v1.0.0`）などの節目に付けることが多い。
    - 軽量タグ（lightweight）とアノテーションタグ（annotated）の2種類があり、リリース用途にはメッセージや署名を持てるアノテーションタグが推奨される。
    - `git tag <name>` で作成、`git push origin <name>` でリモートに反映する。
- 差分を確認する（`git diff`）
    - 変更の差分を表示するコマンド。何をどう変えたかをコミット前に確認できる。
    - `git diff`: 作業ツリー vs ステージングの差分。
    - `git diff --staged`: ステージング vs 直前のコミットの差分。
    - `git diff <commit> <commit>`: 指定した2つのコミット間の差分。
- ブランチ命名の習慣
    - 役割が一目で分かるよう、用途を表すプレフィックスを付けて命名するのが一般的。
    - プレフィックス
        - `feature/`（機能追加）、`bugfix/` または `fix/`（バグ修正）、`hotfix/`（緊急修正）、`release/`（リリース準備）などを用途に応じて使い分ける。
        - 例: `feature/login-form`。単語はハイフン区切り（kebab-case）、課題番号を含める運用も多い（例: `feature/123-login-form`）。
- ブランチ戦略
    - ブランチの運用ルールをあらかじめ決めておく方針。チームの規模やリリース頻度に応じて選ぶ。代表的な戦略は次の2つ。
        - Git Flow: `main` / `develop` / `feature` / `release` / `hotfix` の5種類のブランチを使い分ける。
        - GitHub Flow: `main` と `feature` ブランチのみのシンプルな運用。
- エイリアス（別名）を設定する（`git config`）
    - よく使うコマンドやオプションに短い別名を付ける機能。タイプ量を減らせる。
    - `git config --global alias.<別名> <コマンド>` で登録し、設定は `~/.gitconfig` に保存される。
    - 登録後は `git st` のように使える。`git config --global --get-regexp alias` で一覧を確認できる。
    - 設定例:

```sh
git config --global alias.st status
git config --global alias.br branch
git config --global alias.co switch
git config --global alias.lg "log --oneline --graph --all"
```

## 9. 実践

- ねらい
    - git-demo-repository を使い、main と feature ブランチによる開発フローを体験する。
    - 実際に手を動かして、主要な git コマンドを身につける。
    - 対象コマンド（前半）: `clone` / `switch` / `branch` / `status` / `add` / `commit`
    - 対象コマンド（後半）: `push` / `fetch` / `pull` / `log` / `diff` / `stash`
    - あわせて `.gitignore` による除外設定と、コンフリクト解消も体験する
- 作業場所
    - git-demo-repository の中に自分用のフォルダ（例: `users/<自分の名前>/`）を作り、その中だけで作業する。
    - 各自のフォルダを分けることで、他の人の変更と衝突せずに練習を進められる。
- 事前準備
    - リポジトリを手元に取得する（`git clone <url>`）。
    - 初回のみ、コミットに使う名前とメールを設定する（`git config --global user.name` / `user.email`）。
- `hello.md` の初期内容（サンプル）
    - `users/<自分の名前>/hello.md` を次の内容で作成する。1行目は演習4で編集する。

```markdown
# <自分の名前> の Git 練習ページ

Git の基本操作を練習するためのメモです。

## 今日やること
- ブランチを作ってコミットする
- リモートにプッシュする

## メモ
（ここに自由に追記する）
```

### 演習1: 基本フローを一通り通す
1. `git branch -a` で既存のブランチ一覧を確認する。
2. main を最新にする（`git switch main` → `git pull`）。
3. 作業用ブランチを作って切り替える（`git switch -c feature/<name>-practice`）。
4. `users/<自分の名前>/` フォルダを作り、上記サンプルの `hello.md` を追加する。
5. `git status` で変更状態を、`git diff` で変更内容を確認する。
6. `git add users/<自分の名前>/` でステージングする。
7. もう一度 `git status` を実行し、staged になったことを確認する。
8. `git commit -m "..."` で記録する。4〜8を必要なだけ繰り返す。
9. `git log --oneline --graph` でコミット履歴を確認する。
10. `git push -u origin feature/<name>-practice` でリモートに反映する。
11. ホスティングサービス上で、push した feature ブランチから main 宛てのプルリクエストを作成する（画面操作の詳細は省略）。
12. レビューを受け、承認されたら Squashマージで main に統合する。
13. ローカルの main を最新化する（`git switch main` → `git pull`）。
14. 役目を終えた feature ブランチを削除する。
    - ローカル: `git branch -d feature/<name>-practice`
    - リモート: `git push origin --delete feature/<name>-practice`

### 演習2: 追跡させないファイルを設定する（.gitignore）
1. 動作確認用に、`users/<自分の名前>/secret.txt` を作成する。
2. `git status` で secret.txt が untracked として表示されることを確認する。
3. 同じフォルダに `.gitignore` を作成し、無視したいパターンを書く（例: `secret.txt`、`*.log`）。
4. もう一度 `git status` を実行し、secret.txt が一覧から消えたことを確認する。
5. `.gitignore` 自体は追跡対象にする（`git add users/<自分の名前>/.gitignore` → `git commit`）。

### 演習3: リモートの更新を取り込む（fetch / pull）
- `git fetch` でリモートの最新情報だけを取得する。作業ツリーは変わらない。
- `git log --oneline origin/main` で、取得した remote-tracking branch の状態を確認する。
- `git switch main` → `git pull` で、その更新を手元の main に取り込む。

### 演習4: 作業を一時退避する（stash）
- ファイルを編集した状態で `git stash` を実行し、作業ツリーをクリーンに戻す。
- `git switch` で別ブランチに切り替え、`git status` がクリーンなことを確認する。
- 元のブランチに戻り、`git stash pop` で退避した変更を復元する。

### 演習5: コンフリクトを起こして解消する
1. feature ブランチで `users/<自分の名前>/hello.md` の1行目を編集し、commit する。
2. `git switch main` で切り替え、同じファイルの同じ1行目を別の内容に編集し、commit する。
3. feature に戻り（`git switch feature/<name>-practice`）、`git merge main` を実行するとコンフリクトが発生する。
4. `git status` で競合しているファイルを確認する。
5. 競合マーカー（`<<<<<<<` / `=======` / `>>>>>>>`）を見て、正しい内容に手で編集する。
6. `git add <file>` でステージングし、`git commit` でマージを完了する。
7. `git log --oneline --graph` で分岐とマージの履歴を確認する。


実践の流れ
既存リポジトリをcloneして実装してmainを更新するまで。
使用するリポジトリは、プロジェクトのものではなく、練習用のリポジトリを用意して使うことを推奨。

- config
    - username, emailなどの設定手順の説明
- clone
    - 実際にcloneする
    - cloneとは何かを図解する
    - 通常のフォルダとリポジトリの違いについて簡単に解説
　　 - 新しくリポジトリを作成する方法も補足として解説 　
- branch
    - branch で現在のブランチに何があるか確認する
    - ブランチの一般的な命名規則について解説
    - branch の種類について解説
        - リモートブランチ、ローカルブランチ、リモートトラッキングブランチ
- switch
    - switch で作業ブランチに切り替える
    - switch と checkout の違いについて補足する
    - 新しいブランチを作りながら切り替える `-c` オプションを解説する
- status
    - 実際に何らかのファイルを変更し、`git status` の出力がどう変わるかを確認する
    - `git status` の出力メッセージに含まれる次のアクションのヒントを読み解く練習をする
    - untracked / modified / staged / deleted といった各ステータスの意味を整理して解説する
    - `git status -s` の短縮表示オプションも紹介し、出力の読み方を解説する
- ignore
    - .gitignore を作成して無視ルールを記述する
    - どのパターンがどのファイルにマッチするかを実際のファイルで確認する
    - すでに追跡中のファイルを後から無視する方法（`git rm --cached`）を補足する
- stash
    - `git stash` で未コミットの変更を一時退避し、作業ツリーをクリーンにする
    - `git stash list` で退避済みスタックの一覧を確認する
    - `git stash pop` で最新の退避内容を復元し、スタックから削除する
    - `git stash drop` で不要になった退避内容を削除する方法を解説する
- add
    - `git add <file>` で特定ファイルをステージングする基本操作を解説する
    - `git add .` でカレントディレクトリ以下の変更をまとめてステージングする方法を紹介する
    - `git add -p` で変更をハンクごとに選択してステージングする対話モードを解説する
    - ステージングエリア（インデックス）の概念を図解し、なぜワンクッション置く設計なのかを再度整理して説明する
- diff
    - `git diff` でワーキングツリーとステージングエリアの差分を確認する
    - `git diff --staged` でステージングエリアと最新コミットの差分を確認する
    - `git diff <branch1>..<branch2>` でブランチ間の差分を比較する方法を解説する
    - 差分の読み方（`+` / `-` の意味、ハンク単位の見方）を整理して解説する
- restore
    - `git restore <file>` でワーキングツリーの変更を直前のコミット状態に戻す方法を解説する
    - `git restore --staged <file>` でステージングエリアから変更を取り消す（アンステージする）方法を解説する
- log
    - `git log` でコミット履歴を一覧表示し、各コミットのハッシュ・作者・日時・メッセージの見方を解説する
    - `git log --oneline` で履歴を1行ずつ簡潔に表示するオプションを紹介する
    - `git log --oneline --graph` でブランチの分岐・マージを視覚的に確認する方法を解説する
    - `git log -p` で各コミットの差分を合わせて表示する方法を紹介する
    - `git log --author` や `--since` / `--until` など、絞り込みオプションの使い方を解説する
- commit
    - `git commit -m "<メッセージ>"` でステージングエリアの内容をコミットする基本操作を解説する
    - コミットメッセージの書き方（件名・本文の構成、Conventional Commits など）のベストプラクティスを紹介する
    - `git commit --amend` で直前のコミットメッセージや内容を修正する方法を解説する
        - 簡単なvi操作の解説。`i` で挿入モード、`Esc` で挿入モードを抜ける、`:wq` で保存して終了、`:q!` で変更を破棄して終了する方法を補足する。
    - `git commit -v` でコミット前に差分を確認しながらメッセージを記述する方法を紹介する
- HEADとhead
    - HEAD が指すものを図解
    - `git log --oneline` の出力で HEAD の位置を確認する練習をする
- 再度ファイルの変更からコミットまで行うadd & commit
- push
    - pushの図解
- PR作成
    - 作成方法は各ホスティンングサービスに委ね、PRを使う理由、できること、PR の基本的な概念（feature ブランチ → base ブランチへの提案）を図解し、ブランチからPRが作成されることなどを説明
- リモート上でのマージ
    - マージの種類について図解
    - マージ後のブランチの状態（リモートでマージされた後、ローカルにはまだ古いブランチが残っている点）を説明する
    - コンフリクトが発生する仕組みを図解
    - コンフリクトの対処方法を解説
        - ローカルリポジトリ上でコンフリクトを解消する手順を、実際のコマンド操作（`git fetch`・`git merge` または `git pull` でコンフリクトを発生させ、ファイルを編集後 `git add` → `git commit` で完了するまで）を追って解説する。実践ではやらない、補足として説明。
        - コンフリクトマーカー（`<<<<<<<` / `=======` / `>>>>>>>`）の読み方と、どちらの変更を採用するかの判断基準を説明する
       
- fetch
    - fetchとは何かを図解
    - origin/main などリモートトラッキングブランチが更新される仕組みを解説する
    - `git fetch` 後に `git log origin/main` や `git diff origin/main` でリモートとの差分を確認する手順を紹介する

    - `git reflog` を使って誤って `--hard` リセットした際の復元手順を補足する
